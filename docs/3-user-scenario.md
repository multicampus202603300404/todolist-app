## US-01

### 신규 사용자 회원가입 및 이메일 인증

**페르소나**
이름: 박지우 / 나이: 17세 / 직업: 고등학교 2학년 재학생
스마트폰을 주로 사용하는 디지털 네이티브 세대로, 수행평가와 시험 준비 일정을 체계적으로 관리하고 싶어 앱을 처음 접하게 되었다. 기존에는 메모 앱에 일정을 기록했으나 마감일 추적이 어려웠다.

**사전 조건**
- 사용자는 앱에 계정이 없는 신규 방문자다.
- 회원가입 화면(SCR-02)에 진입한 상태다.
- 유효한 이메일 계정(지우가 실제로 접근 가능한 수신함)을 보유하고 있다.

**목표**
이메일 인증까지 완료하여 로그인 가능한 계정을 만든다.

**시나리오 흐름**

| 단계 | 구분 | 내용 |
|------|------|------|
| 1 | 사용자 행동 | 이메일 입력란에 `jiwoo.park@gmail.com`을 입력한다. |
| 2 | 사용자 행동 | 비밀번호 입력란에 `Study2026!` (대문자 S, 소문자 tudy, 숫자 2026, 특수문자 !)을 입력한다. |
| 3 | 사용자 행동 | 비밀번호 확인 입력란에 동일한 값 `Study2026!`을 다시 입력한다. |
| 4 | 사용자 행동 | "회원가입" 버튼을 누른다. |
| 5 | 시스템 반응 | 이메일 형식 유효성 검사를 통과한다. |
| 6 | 시스템 반응 | 비밀번호 정책(8~20자, 대문자·소문자·숫자·특수문자 각 1자 이상) 검사를 통과한다. |
| 7 | 시스템 반응 | 이메일 중복 여부를 확인한다. 신규 이메일이므로 중복 없음을 확인한다. |
| 8 | 시스템 반응 | 계정을 생성하고 `jiwoo.park@gmail.com`으로 이메일 인증 링크를 발송한다. |
| 9 | 시스템 반응 | "인증 이메일을 발송했습니다. 이메일을 확인하여 인증을 완료해 주세요." 안내 메시지를 표시한다. |
| 10 | 사용자 행동 | 박지우는 이메일 수신함을 열어 수신된 인증 메일의 인증 링크를 클릭한다. |
| 11 | 시스템 반응 | 이메일 인증이 완료된다. 계정이 활성화된다. |
| 12 | 시스템 반응 | "이메일 인증이 완료되었습니다. 로그인 해주세요." 메시지와 함께 로그인 화면(SCR-01)으로 이동한다. |

**대안 흐름**

| 케이스 | 상황 | 시스템 반응 |
|--------|------|-----------|
| 2a | 비밀번호가 `study2026` (특수문자, 대문자 없음)인 경우 | "비밀번호는 8~20자이며 대문자, 소문자, 숫자, 특수문자를 각각 1자 이상 포함해야 합니다" 오류를 표시하고 4단계로 돌아간다. |
| 4a | 이미 가입된 이메일 주소를 입력한 경우 | "이미 사용 중인 이메일입니다" 오류를 표시하고(HTTP 409) 1단계로 돌아간다. |
| 4b | 이메일 형식이 `jiwoo@` 처럼 올바르지 않은 경우 | "올바른 이메일 형식을 입력해주세요" 오류를 표시하고(HTTP 400) 1단계로 돌아간다. |
| 10a | 이메일 인증 링크가 만료된 경우 | "인증 링크가 만료되었습니다. 인증 이메일을 재발송하시겠습니까?" 안내를 표시한다. |
| 12a | 이메일 인증을 완료하지 않고 로그인을 시도한 경우 | 로그인이 차단되고 "이메일 인증을 먼저 완료해주세요." 안내를 표시한다. |

**사후 조건**
- 박지우의 계정이 활성화 상태로 DB에 저장되어 있다.
- 로그인이 가능한 상태가 된다.
- 할일 데이터는 아직 존재하지 않는다.

**참조**
- 기능 ID: F-A-04, F-A-05
- 유스케이스: UC-A-01
- 비즈니스 규칙: BR-U-01, BR-U-04
- 화면: SCR-02

## US-02

### 로그인 정상 처리 및 계정 잠금

**페르소나**
이름: 김민준 / 나이: 34세 / 직업: IT 기업 소프트웨어 엔지니어
업무 마감 일정과 사이드 프로젝트 진행 현황을 함께 관리하기 위해 앱을 사용하고 있다. 여러 서비스에 다른 비밀번호를 사용하는 습관이 있어 종종 비밀번호를 혼동한다.

**사전 조건**
- 김민준은 이미 이메일 인증까지 완료된 계정(`minjun.kim@kakao.com`)을 보유하고 있다.
- 로그인 화면(SCR-01)에 진입한 상태다.

**목표**
올바른 인증 정보로 로그인하여 메인 화면으로 진입한다. 또한 비밀번호를 5회 연속 오입력했을 때 계정이 잠기는 보안 정책을 경험한다.

**시나리오 흐름 (A — 정상 로그인)**

| 단계 | 구분 | 내용 |
|------|------|------|
| 1 | 사용자 행동 | 이메일 입력란에 `minjun.kim@kakao.com`을 입력한다. |
| 2 | 사용자 행동 | 비밀번호 입력란에 `Dev!2026Mk`를 입력한다. |
| 3 | 사용자 행동 | "로그인" 버튼을 누른다. |
| 4 | 시스템 반응 | 이메일과 비밀번호가 일치함을 확인한다. |
| 5 | 시스템 반응 | JWT 액세스 토큰(유효기간 30분)과 Refresh Token(유효기간 7일)을 발급한다. Refresh Token은 httpOnly Cookie에 저장된다. |
| 6 | 시스템 반응 | 메인 할일 목록 화면(SCR-03)으로 이동한다. |

**시나리오 흐름 (B — 계정 잠금)**

| 단계 | 구분 | 내용 |
|------|------|------|
| 1 | 사용자 행동 | 이메일 `minjun.kim@kakao.com`에 잘못된 비밀번호 `dev2026mk`를 입력하고 "로그인"을 누른다. |
| 2 | 시스템 반응 | "이메일 또는 비밀번호가 올바르지 않습니다" 오류를 표시한다. (로그인 실패 1회 누적) |
| 3 | 사용자 행동 | 잘못된 비밀번호로 2, 3, 4회를 연속으로 더 시도한다. |
| 4 | 시스템 반응 | 매회 동일한 오류를 표시하며 실패 횟수를 누적한다. (4회 누적) |
| 5 | 사용자 행동 | 5번째로 틀린 비밀번호를 입력하고 "로그인"을 누른다. |
| 6 | 시스템 반응 | 5회 연속 실패를 감지한다. 계정을 잠금 처리한다(HTTP 423). |
| 7 | 시스템 반응 | "로그인 시도가 5회 초과되어 계정이 잠겼습니다. 이메일로 발송된 안내를 확인해주세요." 메시지를 표시한다. |

**대안 흐름**

| 케이스 | 상황 | 시스템 반응 |
|--------|------|-----------|
| 3a | 이메일 인증을 완료하지 않은 계정으로 로그인을 시도한 경우 | 로그인이 차단되고 이메일 인증 완료 요청 안내를 표시한다. |
| 4a | 계정이 잠긴 상태에서 올바른 비밀번호로 로그인을 시도한 경우 | HTTP 423을 반환하며 계정 잠금 안내 메시지를 표시한다. |

**사후 조건**
- 흐름 A: 사용자 세션이 시작되고, 메인 화면에서 자신의 할일 목록을 볼 수 있는 상태다.
- 흐름 B: 계정이 잠금 상태가 되어 추가 로그인이 차단된 상태다.

**참조**
- 기능 ID: F-A-01, F-A-03
- 유스케이스: UC-A-02
- 비즈니스 규칙: BR-U-02
- 화면: SCR-01
- NFR: NFR-S-02, NFR-S-03

## US-03

### 첫 할일 등록

**페르소나**
이름: 이수아 / 나이: 22세 / 직업: 대학교 3학년 재학생 (경영학과)
오늘 처음 이메일 인증을 완료하고 로그인에 성공했다. 다음 달 있는 기말고사 준비 일정을 앱에 등록해 관리하고자 한다.

**사전 조건**
- 이수아는 이메일 인증이 완료된 계정으로 로그인한 상태다.
- 메인 목록 화면(SCR-03)이 표시되어 있고, 등록된 할일이 한 건도 없어 빈 목록 상태다.

**목표**
"경영통계학 기말고사 준비"라는 첫 번째 할일을 시작일과 종료일을 설정하여 등록한다.

**시나리오 흐름**

| 단계 | 구분 | 내용 |
|------|------|------|
| 1 | 사용자 행동 | 메인 화면의 "+" 또는 "할일 추가" 버튼을 눌러 할일 등록 화면(SCR-04)으로 이동한다. |
| 2 | 사용자 행동 | 제목 입력란에 `경영통계학 기말고사 준비`를 입력한다. |
| 3 | 사용자 행동 | 설명 입력란에 `챕터 5~10 범위, 교수님 강조 문제 위주로 복습할 것`을 입력한다. |
| 4 | 사용자 행동 | 시작일을 `2026-05-20`으로 설정한다. |
| 5 | 사용자 행동 | 종료일을 `2026-06-10`으로 설정한다. |
| 6 | 사용자 행동 | "등록" 버튼을 누른다. |
| 7 | 시스템 반응 | 제목(200자 이하), 시작일, 종료일 필수값 검증을 통과한다. |
| 8 | 시스템 반응 | 종료일(6월 10일) >= 시작일(5월 20일) 조건을 만족함을 확인한다. |
| 9 | 시스템 반응 | 할일을 DB에 저장한다. `is_completed = false`, `completed_at = null`로 초기 설정된다. |
| 10 | 시스템 반응 | 오늘 날짜(2026-03-31)가 시작일(2026-05-20)보다 이전이므로 상태가 `UPCOMING`으로 산출된다. |
| 11 | 시스템 반응 | 메인 목록 화면(SCR-03)으로 돌아오며 방금 등록된 할일이 목록에 표시된다. 상태 뱃지에 "시작 전(UPCOMING)"이 표시된다. |

**대안 흐름**

| 케이스 | 상황 | 시스템 반응 |
|--------|------|-----------|
| 2a | 제목을 입력하지 않고 "등록"을 누른 경우 | "제목은 필수이며 200자 이하여야 합니다" 오류를 표시한다. |
| 5a | 종료일을 시작일(5월 20일)보다 이전인 `2026-05-15`로 설정한 경우 | "종료일은 시작일 이후여야 합니다" 오류를 표시하고(HTTP 400) 수정을 요청한다. |
| 4a | 시작일만 입력하고 종료일을 입력하지 않은 경우 | "시작일과 종료일은 필수 항목입니다" 오류를 표시한다. |

**사후 조건**
- 이수아의 계정에 "경영통계학 기말고사 준비" 할일이 1건 등록되어 있다.
- 해당 할일의 초기 상태는 `UPCOMING`이다.
- 메인 화면 목록에서 해당 할일이 확인된다.

**참조**
- 기능 ID: F-T-09
- 유스케이스: UC-T-01
- 비즈니스 규칙: BR-T-01, BR-T-02, BR-T-05
- 화면: SCR-04

## US-04

### 진행 중인 할일 완료 처리 (IN_PROGRESS → COMPLETED_ON_TIME)

**페르소나**
이름: 정다현 / 나이: 31세 / 직업: 콘텐츠 마케터 (프리랜서)
클라이언트 납품 일정을 todolist-app으로 관리하고 있다. 오늘 오후에 "SNS 캠페인 기획서 작성" 작업을 마무리하고 완료 처리를 하려 한다.

**사전 조건**
- 정다현은 로그인한 상태다.
- "SNS 캠페인 기획서 작성" 할일이 등록되어 있다 (시작일: `2026-03-28`, 종료일: `2026-04-02`).
- 오늘 날짜(2026-03-31)가 시작일과 종료일 사이에 있으므로 현재 상태는 `IN_PROGRESS`다.

**목표**
마감일 이전인 오늘 "SNS 캠페인 기획서 작성"을 완료 처리하여 `COMPLETED_ON_TIME` 상태로 전환한다.

**시나리오 흐름**

| 단계 | 구분 | 내용 |
|------|------|------|
| 1 | 사용자 행동 | 메인 목록 화면(SCR-03)에서 "SNS 캠페인 기획서 작성" 항목을 찾는다. 현재 "진행 중(IN_PROGRESS)" 상태 뱃지가 표시되어 있다. |
| 2 | 사용자 행동 | 해당 항목을 클릭하여 할일 상세 화면(SCR-05)으로 이동한다. |
| 3 | 사용자 행동 | 상세 화면에서 "완료 처리" 버튼을 누른다. |
| 4 | 시스템 반응 | "이 할일을 완료 처리하시겠습니까?" 확인 다이얼로그를 표시한다. |

## US-10

### 완료된 할일 삭제 (완료 항목 정리)

**페르소나**
이름: 송지훈 / 나이: 26세 / 직업: 스타트업 사회초년생 (입사 6개월 차)
지난달 완료한 업무 할일들이 목록에 남아 있어 화면이 복잡해졌다. 더 이상 필요 없는 완료 항목들을 정리하고자 한다.

**사전 조건**
- 송지훈은 로그인한 상태다.
- COMPLETED_ON_TIME 상태의 할일 "신규 서비스 기획안 제출"이 목록에 있다.
- 해당 할일의 `is_completed = true`이며 완료 일시가 기록되어 있다.

**목표**
"신규 서비스 기획안 제출" 할일을 영구 삭제하여 목록을 정리한다.

**시나리오 흐름**

| 단계 | 구분 | 내용 |
|------|------|------|
| 1 | 사용자 행동 | 메인 목록 화면(SCR-03)에서 "신규 서비스 기획안 제출" 항목을 클릭하여 상세 화면(SCR-05)으로 이동한다. |
| 2 | 사용자 행동 | 상세 화면에서 "삭제" 버튼을 누른다. |
| 3 | 시스템 반응 | "이 할일을 삭제하시겠습니까? 삭제된 데이터는 복구되지 않습니다." 확인 다이얼로그를 표시한다(NFR-D-02). |
| 4 | 사용자 행동 | "삭제" 버튼을 눌러 삭제를 최종 확인한다. |
| 5 | 시스템 반응 | 소유권 검증 — 해당 할일의 `user_id`가 송지훈의 ID와 일치함을 확인한다. |
| 6 | 시스템 반응 | 해당 할일을 DB에서 영구 삭제한다. |
| 7 | 시스템 반응 | 메인 목록 화면(SCR-03)으로 돌아오며 삭제된 항목이 목록에서 제거되어 있다. "할일이 삭제되었습니다." 토스트 메시지를 표시한다. |

**대안 흐름**

| 케이스 | 상황 | 시스템 반응 |
|--------|------|-----------|
| 4a | 확인 다이얼로그에서 "취소"를 누른 경우 | 삭제가 중단되고 상세 화면(SCR-05)을 그대로 유지한다. |
| 2a | 타인의 할일 ID로 삭제를 시도하는 경우(API 직접 호출) | 소유권 검증에 실패하여 "할일을 찾을 수 없습니다"를 반환한다(HTTP 404). |
| 2b | 존재하지 않는 할일 ID로 삭제를 시도하는 경우 | "할일을 찾을 수 없습니다"를 반환한다(HTTP 404). |

**사후 조건**

- 목록에서 해당 항목이 보이지 않는다.
- 통계 수치가 재산출 시 해당 건 제외된 값으로 표시된다.
- 삭제된 데이터는 복구 불가능하다.

## US-11

### 토큰 만료 후 자동 갱신 (Silent Refresh)

**페르소나**
이름: 임태양 / 나이: 41세 / 직업: 중견기업 팀장 (영업팀)
퇴근 후 늦은 시간에 앱을 열어 내일 일정을 정리하는 습관이 있다. 오늘은 점심에 앱에 접속했다가 브라우저를 닫지 않고 6시간 후 다시 사용하려 한다.

**사전 조건**
- 임태양은 6시간 전 로그인에 성공했고, 당시 JWT 액세스 토큰(유효기간 30분)이 발급되었다.
- Refresh Token(유효기간 7일)은 httpOnly Cookie에 저장된 상태다.
- 현재 액세스 토큰은 만료된 상태다.
- 브라우저 탭은 열려 있고 메인 화면(SCR-03)이 표시된 상태다.

**목표**
액세스 토큰이 만료되었음에도 사용자가 로그아웃 없이 원활하게 할일을 계속 사용할 수 있도록 한다.

**시나리오 흐름**

| 단계 | 구분 | 내용 |
|------|------|------|
| 1 | 사용자 행동 | 임태양이 화면을 다시 보고 새로운 할일 "내일 거래처 미팅 준비"를 등록하기 위해 "+" 버튼을 누른다. |
| 2 | 시스템 반응 | 클라이언트가 할일 등록 API를 호출하려 하지만, 저장된 액세스 토큰이 만료 상태임을 감지한다. |
| 3 | 시스템 반응 | 사용자에게 인지시키지 않고(사용자 개입 없이) httpOnly Cookie에 저장된 Refresh Token으로 토큰 갱신 API를 자동 호출한다 (Silent Refresh). |
| 4 | 시스템 반응 | 서버가 Refresh Token의 유효성을 검증한다. Refresh Token이 유효하고(7일 내) 폐기되지 않았음을 확인한다. |
| 5 | 시스템 반응 | 새로운 JWT 액세스 토큰(유효기간 30분)을 발급한다. |
| 6 | 시스템 반응 | 클라이언트가 새로 발급된 액세스 토큰으로 원래 요청(할일 등록)을 재시도한다. |
| 7 | 시스템 반응 | 할일 등록이 정상 처리되고 "내일 거래처 미팅 준비" 항목이 목록에 추가된다. |
| 8 | 사용자 행동 | 임태양은 별도의 로그인 절차 없이 자연스럽게 할일 등록이 완료된 것을 확인한다. |

**대안 흐름**

| 케이스 | 상황 | 시스템 반응 |
|--------|------|-----------|
| 4a | Refresh Token도 만료(7일 초과)된 경우 | 토큰 갱신이 실패하고 클라이언트는 사용자를 로그인 화면(SCR-01)으로 리다이렉트한다. "세션이 만료되었습니다. 다시 로그인해주세요." 메시지를 표시한다. |
| 4b | Refresh Token이 서버에서 폐기(로그아웃 처리 등)된 경우 | 토큰 갱신이 실패하고 로그인 화면으로 이동시킨다. |

**사후 조건**
- 임태양의 세션이 연장되어 새 액세스 토큰이 발급된 상태다.
- 할일 등록이 정상 완료되었다.
- 사용자 경험 관점에서 중단 없이 작업이 이어졌다.

**참조**
- 유스케이스: UC-A-02
- NFR: NFR-S-02, NFR-S-03, NFR-S-04
- 관련 기능: F-A-01

## US-12

### 모바일에서 빠른 완료 처리

**페르소나**
이름: 배하린 / 나이: 16세 / 직업: 중학교 3학년 재학생
스마트폰으로 일상적인 학교 준비물과 숙제 일정을 관리하고 있다. 상세 화면으로 이동하지 않고 목록 화면에서 바로 완료 체크를 하고 싶어 한다.

**사전 조건**
- 배하린은 스마트폰 모바일 웹 브라우저에서 로그인한 상태다.
- 메인 목록 화면(SCR-03)이 반응형 모바일 레이아웃으로 표시되어 있다.
- IN_PROGRESS 상태의 할일 "수학 숙제 제출"이 목록에 있다 (시작일: `2026-03-30`, 종료일: `2026-03-31`).

**목표**
할일 상세 화면으로 이동하지 않고, 목록 화면의 체크박스를 직접 탭하여 "수학 숙제 제출"을 완료 처리한다.

**시나리오 흐름**

| 단계 | 구분 | 내용 |
|------|------|------|
| 1 | 사용자 행동 | 배하린이 모바일 브라우저에서 메인 목록 화면(SCR-03)을 연다. 할일 항목 목록이 카드 형태로 표시된다. |
| 2 | 사용자 행동 | "수학 숙제 제출" 항목 왼쪽의 체크박스를 손가락으로 탭한다. |
| 3 | 시스템 반응 | 체크박스 탭 이벤트를 감지하고 완료 처리 API를 호출한다. |
| 4 | 시스템 반응 | `is_completed = true`로 변경하고 `completed_at = 2026-03-31T(현재시각)`을 자동 기록한다. |
| 5 | 시스템 반응 | `completed_at(3월 31일) <= end_date(3월 31일)` 조건을 만족하므로 상태가 `COMPLETED_ON_TIME`으로 산출된다. |
| 6 | 시스템 반응 | 목록 화면에서 해당 항목의 체크박스가 체크된 상태로 전환되고, 상태 뱃지가 "성공적으로 완료(COMPLETED_ON_TIME)"로 업데이트된다. 별도 화면 이동 없이 현재 목록 화면을 유지한다. |
| 7 | 사용자 행동 | 배하린은 다른 항목들도 연속으로 체크박스를 탭하며 빠르게 완료 처리를 이어간다. |

**대안 흐름**

| 케이스 | 상황 | 시스템 반응 |
|--------|------|-----------|
| 2a | 이미 완료(체크)된 항목의 체크박스를 다시 탭하여 완료 해제를 시도하는 경우 | "이 할일의 완료를 취소하시겠습니까?" 간단한 확인 메시지를 표시한다. 확인 시 `is_completed = false`, `completed_at = null`로 초기화한다(UC-T-07). |
| 3a | 네트워크 불안정으로 API 호출이 실패한 경우 | 체크박스가 이전 상태로 롤백되고 "요청에 실패했습니다. 다시 시도해주세요." 토스트 메시지를 표시한다. |
| 1a | 반응형 레이아웃이 아닌 데스크탑 화면에서 접근한 경우 | 동일한 체크박스 방식이 데스크탑 레이아웃에서도 동일하게 동작한다 (반응형 동일 기능). |

**사후 조건**
- "수학 숙제 제출" 할일의 `is_completed = true`, `completed_at`에 오늘 일시가 기록되어 있다.
- 상태가 `COMPLETED_ON_TIME`으로 표시된다.
- 사용자는 목록 화면(SCR-03)에 그대로 있다 (화면 전환 없음).
- 모바일 환경에서의 빠른 인터랙션이 보장된다.

**참조**
- 기능 ID: F-T-07
- 유스케이스: UC-T-06, UC-T-07
- 비즈니스 규칙: BR-T-03, BR-T-04, BR-T-05
- 화면: SCR-03 (반응형 모바일 레이아웃)
